# 18. Parent-Child Chunking 分块策略开发方案

## 1. 背景与目标

### 1.1 现有分块策略的局限

当前系统提供两种分块策略：

- **paragraph（字符分块）**：按段落/句子切分后按 `maxSegmentLength` 合并，粒度均匀但缺乏语义层次；
- **chapter（章节分块）**：识别文档章节结构，适合结构化文档，但依赖正则或字体元数据，泛化能力有限。

两种策略都是「平铺单层」的分块方式，存在一个根本性矛盾：

| 目标 | 要求 |
|------|------|
| 向量检索命中精准 | 块越小越好，语义集中，噪声少 |
| LLM 理解上下文丰富 | 块越大越好，包含足够背景信息 |

无论把块切多大还是多小，单层策略都无法同时满足这两个目标。

### 1.2 解决方案：Parent-Child Chunking

Parent-Child Chunking（父子分块，又称 Small-to-Big Retrieval）是目前 RAG 工程中被广泛验证的最优实践之一。其核心思想是：

- **Child 块（子块）**：粒度小（约 200~400 字），用于向量化与语义检索，保证检索的精准性；
- **Parent 块（父块）**：粒度大（约 1000~2000 字），仅存储于关系数据库，用于向 LLM 提供丰富的上下文。

检索时，系统先用子块做向量匹配，命中后自动获取对应的父块内容喂给 LLM，兼顾了检索精度与回答质量。

### 1.3 开发目标

1. 在现有分块架构上新增 `parent_child` 分块策略；
2. 将 `parent_child` 设为新建知识库文档时的**默认分块策略**；
3. 改造 RAG 检索链路，支持「子块命中 → 父块上下文」的检索扩展逻辑；
4. 前端配置界面同步支持 Parent-Child 参数配置；
5. 已有文档保持原有策略不变，新上传文档默认使用新策略。

## 2. 核心概念

### 2.1 分块层次示意

```
原始文档（3000 字）
│
├─ Parent 块 1（约 1200 字）
│   ├─ Child 块 1-1（约 300 字）  ← 存入 Qdrant，用于向量检索
│   ├─ Child 块 1-2（约 300 字）  ← 存入 Qdrant，用于向量检索
│   ├─ Child 块 1-3（约 300 字）  ← 存入 Qdrant，用于向量检索
│   └─ Child 块 1-4（约 300 字）  ← 存入 Qdrant，用于向量检索
│
├─ Parent 块 2（约 1200 字）
│   ├─ Child 块 2-1（约 300 字）
│   ├─ Child 块 2-2（约 300 字）
│   └─ Child 块 2-3（约 300 字）
│
└─ Parent 块 3（约 600 字）
    ├─ Child 块 3-1（约 300 字）
    └─ Child 块 3-2（约 300 字）
```

### 2.2 存储分工

| 数据层 | 父块（Parent） | 子块（Child） |
|--------|---------------|--------------|
| MySQL（`dataset_document_segment`） | ✅ 存储（完整内容） | ✅ 存储（内容 + `parent_segment_id`） |
| Qdrant（向量库） | ❌ 不存储 | ✅ 向量化存储（`vector_id` 非空） |

### 2.3 检索流程变化

**改造前（当前）**：
```
用户问题 → Qdrant 向量检索 → 召回分段内容 → 拼接 Prompt → LLM
```

**改造后**：
```
用户问题 → Qdrant 向量检索（子块）→ 查询 parent_segment_id → 
          从 MySQL 获取父块内容 → 去重 → 拼接 Prompt → LLM
```

## 4. 数据库改造

### 4.1 `dataset_document` 表新增字段

```sql
-- doc/sql/migration_19_parent_child_chunking.sql

ALTER TABLE `dataset_document`
    ADD COLUMN `parent_chunk_size` INT NOT NULL DEFAULT 1200
        COMMENT 'Parent 块最大长度（仅 parent_child 策略使用）'
        AFTER `overlap_length`;

-- 将默认分块策略改为 parent_child（仅影响新创建记录，存量不变）
-- 应用层通过代码默认值控制，此处仅作注释说明
```

### 4.2 `dataset_document_segment` 表新增字段

```sql
ALTER TABLE `dataset_document_segment`
    ADD COLUMN `segment_type` TINYINT NOT NULL DEFAULT 0
        COMMENT '分段类型：0=普通（legacy），1=父块，2=子块'
        AFTER `position`,
    ADD COLUMN `parent_segment_id` BIGINT DEFAULT NULL
        COMMENT '父块 segment_id，仅 segment_type=2 时有值'
        AFTER `segment_type`,
    ADD INDEX `idx_parent_segment_id` (`parent_segment_id`);
```

### 4.3 字段说明

`segment_type` 使用整数代替字符串枚举，节省存储并便于索引：

| 值 | 含义 |
|----|------|
| `0` | 普通分段（paragraph / chapter 策略，历史数据） |
| `1` | 父块（parent_child 策略，不进向量库） |
| `2` | 子块（parent_child 策略，进向量库，`parent_segment_id` 指向父块） |


**`SplitMode.java`** 新增枚举值：

```java
PARENT_CHILD_SPLITTER
```

**`DatasetDocumentSegment.java`** 新增字段（对应数据库新列）：

```java
@TableField("segment_type")
private Integer segmentType;

@TableField("parent_segment_id")
private Long parentSegmentId;
```

### 5.2 ParentChildSplitter 设计

新建文件：`api/src/main/java/com/yxboot/ai/document/splitter/ParentChildSplitter.java`

**构造参数**：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `parentChunkSize` | int | 1200 | 父块最大字符数 |
| `childChunkSize` | int | 300 | 子块最大字符数 |
| `childOverlapSize` | int | 50 | 子块重叠字符数 |

**核心算法**：

```
splitText(text):
  1. 用 CharacterSplitter(parentChunkSize, 0) 切出父块列表 parentChunks
  2. 遍历 parentChunks：
     a. 生成父块 DocumentSegment（segmentType=1，id=UUID）
     b. 用 CharacterSplitter(childChunkSize, childOverlapSize) 对父块内容再次切分
     c. 每个子块生成 DocumentSegment（segmentType=2，parentId=父块id，id=UUID）
  3. 按顺序返回所有 DocumentSegment（父块与其子块交替，或统一展平均可）
```

**返回值约定**：返回包含父块和子块的平铺列表 `List<DocumentSegment>`，通过 `segmentType` 区分，调用方负责将父块和子块分开处理。

**父块内容策略**：父块内容为原始大块文本（不截断），子块内容为从父块文本中切出的片段。父块不额外附加子块内容，子块也不重复存储父块原文，两者通过 `parentId` 关联。

**父块 `title` 生成**：取父块内容前 30 字（去除换行）作为标题，与现有 `AbstractSplitter` 的标题生成逻辑保持一致。

### 5.3 AiDocumentProcessingService 新增分支

在 `loadAndSplitDocument` 的 `switch` 语句中新增：

```java
case PARENT_CHILD_SPLITTER -> splitByParentChild(documents, maxSegmentLength, overlapLength, parentChunkSize);
```

新增方法 `splitByParentChild`：

- 接受额外参数 `parentChunkSize`（取自 `DatasetDocument.parentChunkSize`）；
- 实例化 `ParentChildSplitter(parentChunkSize, maxSegmentLength, overlapLength)` 并调用；
- 返回 `List<DocumentSegment>`，包含全部父块和子块。

> **注意**：这里字段语义发生了复用变化：文档表中原有的 `max_segment_length` / `overlap_length` 用于控制**子块**尺寸，新增的 `parent_chunk_size` 控制**父块**尺寸。

### 5.4 DatasetDocumentProcessingApplicationService 改造

**枚举映射**（`convertSegmentMethodToSplitMode`）新增：

```java
case PARENT_CHILD -> SplitMode.PARENT_CHILD_SPLITTER;
```

**向量化逻辑改造**：

当前流程调用 `vectorStoreService.batchCreateSegmentVectors(segments)` 时，新增过滤：

```java
// 只对子块（segmentType=2）和普通块（segmentType=0）进行向量化，跳过父块（segmentType=1）
List<DatasetDocumentSegment> vectorizableSegments = allSegments.stream()
    .filter(s -> s.getSegmentType() != 1)
    .collect(Collectors.toList());
vectorStoreService.batchCreateSegmentVectors(vectorizableSegments);
```

**批量写入分段**：

调用 `segmentService.batchCreateSegments` 时，父块和子块一起写入，`parent_segment_id` 的值需要在写入前解析（因为父块的 `segment_id` 是雪花 ID，在写入数据库后才确定）。处理步骤：

1. 先批量写入所有父块（`segment_type=1`）；
2. 雪花 ID 写入后，将 AI 层 `DocumentSegment.id`（UUID）与数据库 `segment_id` 的映射关系缓存到 Map；
3. 遍历子块，根据 `DocumentSegment.parentId` 查找对应父块的 `segment_id`，填入 `parent_segment_id`；
4. 批量写入所有子块。

### 5.5 DatasetDocumentSegmentService 改造

新增方法 `batchCreateParentChildSegments(List<DocumentSegment>, DatasetDocument)`，封装上述父子块两阶段写入逻辑，减少 `ProcessingApplicationService` 的复杂度。

### 5.6 API 接口改造

**`DatasetDocumentController`** 的 `CreateDocumentRequest` 新增字段：

```java
// 父块大小（仅 parent_child 策略生效）
private Integer parentChunkSize = 1200;
```

**`DatasetDocumentDTO`** 新增字段，方便前端回显：

```java

private Long totalChildSegments;  // 子块数量（方便展示）
```

**默认策略**：在 `DatasetDocumentApplicationService.createDocument` 中，将 `segmentMethod` 的默认值改为 `PARENT_CHILD`：

```java

    ? request.getSegmentMethod() 
    : SegmentMethod.PARENT_CHILD.getValue();
```

## 6. RAG 检索改造

### 6.1 检索链路整体设计

检索入口：`AiVectorRetrieverService.retrieve(query, datasetId, tenantId)`

**改造前**：向量相似度搜索 → 直接返回分段内容。

**改造后**：

```
1. 向量相似度搜索 → 返回 List<AiQueryResult>（子块）
2. 提取 segment_id 列表
3. 查询 dataset_document_segment 表：
   - 若 segment_type=2（子块）：取其 parent_segment_id
   - 若 segment_type=0（普通块）：直接使用自身内容（保持向后兼容）
4. 按 parent_segment_id 列表批量查询父块内容
5. 对父块内容去重（同一父块可能被多个子块命中）
6. 返回去重后的父块内容列表（保留关联的子块相似度分值，取最高值）
```

### 6.2 AiVectorRetrieverService 改造要点

- 依赖注入 `DatasetDocumentSegmentService`；
- 新增私有方法 `expandToParentSegments(List<AiQueryResult>)`，封装父块扩展逻辑；
- `retrieve` 方法在搜索后调用该方法；
- 去重逻辑：以 `parentSegmentId` 为 key，保留相似度最高的得分。

### 6.3 AiQueryResult 调整

`AiQueryResult` 新增可选字段，方便上层理解检索来源：

```java
// 命中的子块 ID（parent_child 策略下填充）
private Long hitChildSegmentId;
// 命中的子块内容（可选，用于调试）
private String hitChildContent;
```

### 6.4 ChatController 调用不变

`ChatController` 调用 `AiVectorRetrieverService.retrieve()` 的方式不需要修改，父块扩展逻辑完全封装在 `AiVectorRetrieverService` 内部，对上层透明。

## 8. 默认策略的影响范围

将 `parent_child` 设为默认分块策略后，对以下场景有影响：

| 场景 | 影响 | 处理方式 |
|------|------|---------|
| 新上传文档 | 使用父子分块 | 直接生效，无历史数据问题 |
| 已有文档 | 保持原策略（paragraph/chapter） | `segment_type=0`，检索时走兼容逻辑 |
| 重新处理文档 | 按当前文档存储的策略重新分块 | 无需修改，复用原有策略字段 |
| RAG 混合检索 | 同一知识库内可能同时存在普通块和子块 | 检索时按 `segment_type` 分支处理，互不干扰 |

## 10. 注意事项与风险

### 10.1 向量 ID 管理

父块不进向量库，其 `vector_id` 应置为 `null`，以避免与 `AiVectorStoreService` 中基于 `vector_id` 的删除逻辑产生混淆。删除文档时，需确认只对 `vector_id != null` 的分段执行向量删除操作（当前代码需检查此逻辑）。

### 10.2 分段数量膨胀

同等文档，Parent-Child 策略产生的分段数量约为单层策略的 **3~5 倍**（父块 + 子块之和）。需关注：

- `dataset_document_segment` 表的数据量增长；
- 向量写入性能（子块数增加，向量化耗时增加）；
- 建议 `documentTaskExecutor` 线程池参数评估是否需要调整。

### 10.3 子块大小默认值选择

当前 `maxSegmentLength` 默认值为 500。Parent-Child 策略中子块推荐值为 200~400。如将默认值从 500 改为 300，需确认不影响其他分块策略的默认行为（建议子块默认值通过 `ParentChildSplitter` 构造参数独立控制，不修改全局默认值）。

### 10.4 向后兼容性

检索层改造后，`segment_type=0`（历史普通块）必须继续正常工作。在 `AiVectorRetrieverService.expandToParentSegments` 中，对 `segment_type=0` 的分段直接使用原始内容，不尝试查找父块，保持行为一致。

### 10.5 父块分页展示

文档详情页当前按 `position` 排序分页展示所有分段。引入父子块后，如果不区分类型，父块会和子块混排，导致展示混乱。需在查询时默认过滤只展示子块（`segment_type IN (0, 2)`），父块单独在新增的「父块视图」中展示。

### 10.6 编辑/删除子块时的父块同步

当前分段编辑会同步更新 Qdrant 向量。对于子块：编辑子块内容 → 重新向量化子块（逻辑不变）。对于父块：父块不在向量库，无需向量操作，但如果用户在 UI 编辑了父块内容，需要考虑是否联动更新旗下子块（当前 Phase 不处理，作为后续演化）。
