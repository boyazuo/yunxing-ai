# 模型配置静态化改造方案

> 编写时间：2026-05-23  
> 状态：已实施

---

## 一、背景与目标

### 1.1 现状

当前系统中，AI 模型的提供商（Provider）和模型（Model）信息通过数据库动态管理，支持按租户独立配置 API Key 和模型选择。整体调用链路如下：

```
租户配置 Provider（API Key）→ 绑定 Model → 应用引用 modelId → ChatController 查 DB → ChatModelRegistry 动态创建 ChatModel
```

知识库向量化链路：

```
Dataset.embeddingModelId → 查 Model 表 → 查 Provider 表 → EmbeddingModelRegistry 动态创建 EmbeddingModel → VectorStoreRegistry
```

### 1.2 问题

这套设计目前带来了以下问题：

1. **过度设计**：目前系统仅支持智谱 AI（ZhiPu）一个提供商，多租户动态模型管理的复杂度远超实际需求
2. **代码冗余**：Registry 类中大量 `computeIfAbsent` + Provider/Model 对象传递，实际上每次调用的参数几乎相同
3. **数据一致性风险**：`AiVectorStoreService` 中存在 `dummyProvider` / `dummyModel` 的 workaround，说明部分场景已经绕过了动态配置
4. **维护负担**：Provider、Model 的 CRUD 接口、Entity、Mapper、Service、DTO 共 10+ 个文件，维护成本高
5. **扩展受限**：数据库中的模型配置实际上无法支持复杂的多提供商路由策略

### 1.3 改造目标

- 移除 `provider` 表和 `model` 表
- 对话模型（Chat Model）和向量模型（Embedding Model）统一在后端配置文件（`application.yml`）中配置
- 租户不再有权配置和选择模型
- 简化 AI 相关的框架、注册表、服务层代码
- 知识库（Dataset）不再绑定 embeddingModelId，统一使用系统配置的向量模型

## 三、配置文件现状问题分析

### 3.1 现有配置的问题全景

改造前，AI 相关配置分散在两处，且存在大量冗余和遗留死代码：

**`application.yml`（全局配置）中的问题：**

```yaml
spring:
  ai:
    model:
      chat: none        # ❌ 补丁配置：强制禁用 Spring AI 自动配置的 ChatModel
      embedding: none   # ❌ 补丁配置：强制禁用 Spring AI 自动配置的 EmbeddingModel
    zhipuai:
      api-key: placeholder  # ❌ 无效占位符，真实 Key 来自数据库
    vectorstore:
      type: none        # ❌ 补丁配置：禁用 Spring AI 自动配置的 VectorStore
      qdrant:           # ✅ 保留：QdrantClient Bean 实际读取此配置
        host: localhost
        port: 6334
```

> **根因**：上述三处 `none` / `placeholder` 是为了绕开 Spring AI 自动配置而打的补丁，因为当前模型实例由数据库动态创建，不走 Spring AI 标准自动配置链路。改造后这些补丁可以全部移除。

**`application-dev.yml`（环境配置）中的问题：**

```yaml
yxboot:
  llm:
    client:               # ❌ 遗留死代码：Spring AI 引入之前的自定义 HTTP 客户端配置
      cache-enabled: true
      max-cache-size: 100
      ...（共 11 行）

    embedding:
      zhipuai:            # ⚠️ 实际在用，但需迁移到新 yxboot.ai 节点
        api-key: 
        model-name: embedding-3
        base-url: https://open.bigmodel.cn/api/paas/v4/embeddings
        embedding-dimension: 2048
        batch-size: 32
      client:             # ❌ 与 yxboot.llm.client 内容完全一致，纯复制粘贴冗余
        cache-enabled: true
        ...（共 11 行）

    vector:
      qdrant:             # ❌ 重复：与 spring.ai.vectorstore.qdrant 配置重叠
        host: localhost
        port: 6333
        grpc-port: 6334
        ...
```

## 四、新架构设计

### 4.1 SpringAIConfig — 统一初始化 AI Bean

改造后，`SpringAIConfig` 负责根据配置文件初始化 ChatModel 和 EmbeddingModel Bean，支持按 `provider` 字段路由到不同实现：

```java
@Configuration
@EnableConfigurationProperties(AiProperties.class)
public class SpringAIConfig {

    // ChatModel Bean（单例）
    @Bean
    public ChatModel chatModel(AiProperties props) {
        AiProperties.ChatConfig cfg = props.getChat();
        return switch (cfg.getProvider().toLowerCase()) {
            case "zhipuai", "zhipu" -> buildZhiPuChatModel(cfg);
            // case "openai" -> buildOpenAiChatModel(cfg);
            default -> throw new IllegalArgumentException("不支持的 Chat 提供商: " + cfg.getProvider());
        };
    }

    // EmbeddingModel Bean（单例）
    @Bean
    public EmbeddingModel embeddingModel(AiProperties props) {
        AiProperties.EmbeddingConfig cfg = props.getEmbedding();
        return switch (cfg.getProvider().toLowerCase()) {
            case "zhipuai", "zhipu" -> buildZhiPuEmbeddingModel(cfg);
            default -> throw new IllegalArgumentException("不支持的 Embedding 提供商: " + cfg.getProvider());
        };
    }

    // QdrantClient Bean（保留）
    @Bean
    @ConditionalOnMissingBean
    public QdrantClient qdrantClient(...) { ... }

    private ChatModel buildZhiPuChatModel(AiProperties.ChatConfig cfg) {
        ZhiPuAiApi api = ZhiPuAiApi.builder().apiKey(cfg.getApiKey()).build();
        ZhiPuAiChatOptions options = ZhiPuAiChatOptions.builder()
                .model(cfg.getModel())
                .temperature(cfg.getTemperature())
                .topP(cfg.getTopP())
                .maxTokens(cfg.getMaxTokens())
                .build();
        return new ZhiPuAiChatModel(api, options);
    }

    private EmbeddingModel buildZhiPuEmbeddingModel(AiProperties.EmbeddingConfig cfg) {
        ZhiPuAiApi api = ZhiPuAiApi.builder().apiKey(cfg.getApiKey()).build();
        ZhiPuAiEmbeddingOptions options = ZhiPuAiEmbeddingOptions.builder()
                .model(cfg.getModel())
                .dimensions(cfg.getDimensions())
                .build();
        return new ZhiPuAiEmbeddingModel(api, MetadataMode.EMBED, options);
    }
}
```

### 4.2 Registry 简化（可选保留或直接移除）

由于 ChatModel 和 EmbeddingModel 已成为单例 Spring Bean，`ChatModelRegistry` 和 `EmbeddingModelRegistry` 失去存在的必要。

**方案 A（推荐）：直接删除两个 Registry 类**，所有需要 ChatModel/EmbeddingModel 的地方直接通过 `@RequiredArgsConstructor` 注入 Spring Bean。

**方案 B：保留 Registry 作为 Bean 的简单包装**（适合未来可能需要多模型路由的场景），但当前阶段过于冗余，不推荐。

### 4.3 VectorStoreRegistry 简化

移除 Provider/Model 参数，直接注入 `EmbeddingModel` Bean：

```java
@Component
@RequiredArgsConstructor
public class VectorStoreRegistry {

    private final QdrantClient qdrantClient;
    private final EmbeddingModel embeddingModel;  // 直接注入 Bean
    private final ConcurrentHashMap<String, VectorStore> storeCache = new ConcurrentHashMap<>();

    // getOrCreate 方法签名简化：移除 Provider、Model 参数
    public VectorStore getOrCreate(Long datasetId, Long tenantId) {
        String collectionName = buildCollectionName(datasetId, tenantId);
        return storeCache.computeIfAbsent(collectionName, k -> createVectorStore(collectionName));
    }

    private VectorStore createVectorStore(String collectionName) {
        return QdrantVectorStore.builder(qdrantClient, embeddingModel)
                .collectionName(collectionName)
                .initializeSchema(true)
                .build();
    }

    // collectionExists、deleteCollection 方法保持不变
}
```

移除所有 `Provider provider, Model model` 参数，同时删除 `dummyProvider()` / `dummyModel()` 方法：

```java
// 改造前

public List<AiQueryResult> similaritySearch(Long datasetId, Long tenantId, ...)
```

所有调用 `vectorStoreRegistry.getOrCreate(datasetId, tenantId, provider, model)` 的地方统一改为 `vectorStoreRegistry.getOrCreate(datasetId, tenantId)`。

### 4.5 AiVectorRetrieverService 简化

移除 `ProviderService` 和 `ModelService` 依赖：

```java
// 改造前
public List<AiQueryResult> retrieve(Long datasetId, String query, ...) {

// 改造后

    return vectorStoreService.similaritySearch(datasetId, dataset.getTenantId(), ...);
}
```

### 4.6 ChatService 简化

直接注入 `ChatModel` Bean，移除 `ChatModelRegistry` 和 `Provider` 参数：

```java
// 改造前
@RequiredArgsConstructor
public class ChatService {
    private final ChatModelRegistry chatModelRegistry;

    public ChatResponseDTO chatCompletion(Provider provider, ChatRequestDTO request) {
        var chatModel = chatModelRegistry.getOrCreate(provider);
        ...
    }
}

// 改造后
@RequiredArgsConstructor
public class ChatService {
    private final ChatModel chatModel;          // 直接注入
    private final MessageService messageService;

    public ChatResponseDTO chatCompletion(ChatRequestDTO request) {
        Prompt prompt = buildPrompt(request);
        ChatResponse response = chatModel.call(prompt);
        ...
    }
}
```

`buildPrompt` 中的 `ZhiPuAiChatOptions` 动态构建仍然保留，支持 per-request 的 temperature / topP / maxTokens 覆盖（来自 AppConfig 的模型参数配置）。

### 4.7 ChatController 简化

移除 `ProviderService` 依赖，移除 `ModelConfig.modelId` / `provider` 字段查询逻辑：

```java
// 改造前：从 AppConfig.models JSON 中按 modelId 查找 Provider
Provider provider = providerService.getProviderByModelId(request.getModelId());
List<ModelConfig> modelConfigs = ...;
ModelConfig modelConfig = modelConfigs.stream()
    .filter(m -> m.getModelId().equals(request.getModelId())).findFirst()...;

// 改造后：AppConfig.models 只存推理参数，直接读取
AppModelConfig modelConfig = appConfigService.getModelConfig(request.getAppId());
if (modelConfig != null) {
    request.setTemperature(modelConfig.getTemperature());
    request.setTopP(modelConfig.getTopP());
    request.setMaxTokens(modelConfig.getMaxTokens());
}
```

同时 `ChatRequestDTO` 移除 `modelId` 字段（不再需要）。

### 4.8 AppConfig.models 字段语义变更

`app_config.models` 字段由原来的「模型选择 + 推理参数」简化为「仅存推理参数」：

```json
// 改造前（含 modelId / provider 等外键引用）
[{
  "id": "cfg_001",
  "modelId": 1234567890,
  "modelName": "glm-4-flash",
  "provider": "zhipuai",
  "temperature": "0.7",
  "topP": "0.95",
  "maxTokens": "4096"
}]

// 改造后（仅推理参数，去除 modelId/provider）
{
  "temperature": 0.7,
  "topP": 0.95,
  "maxTokens": 4096
}
```

`AppConfig` 实体的 `models` 字段可以重命名为 `modelParams`，或直接将字段拆分为独立列（推荐后者，可读性更好）。

> **字段拆分方案（可选，更清晰）**：将 `models` TEXT 字段拆分为 `temperature DOUBLE`、`top_p DOUBLE`、`max_tokens INT` 三个独立字段，移除 `models` 字段。此变更影响 `app_config` 表和 `AppConfig` 实体，是可选的进一步优化。

### 4.9 Dataset 简化

`dataset` 表和 `Dataset` 实体移除 `embedding_model_id` 字段。

`DatasetService.createDataset` 方法签名简化：

5. 确认 API 文档（Swagger）中已无 Provider/Model 相关接口

---

## 七、预期收益

| 维度 | 改造前 | 改造后 |
|---|---|---|
| 数据库表数量 | provider + model（2张表） | 无（移除） |
| AI 相关 Java 文件 | ~20 个 | ~12 个（减少约 40%） |
| Registry 复杂度 | 动态 ConcurrentHashMap 缓存 | 单例 Spring Bean |
| 参数传递链路 | Provider/Model 对象在 5+ 个方法间传递 | 无需传递 |
| 配置管理 | 数据库 CRUD + 前端配置页面 | 单一配置文件 |
| 新增提供商成本 | 后端 Registry + 前端页面 | 仅修改 SpringAIConfig + 配置文件 |

---

## 八、注意事项与风险

1. **前端影响**：如果前端已有「模型管理」或「提供商管理」相关页面，需同步下线对应页面和 API 调用。
2. **AppConfig.models 数据迁移**：若数据库中已有 `app_config.models` 数据，需要在执行 DDL 前做数据迁移，将旧 JSON 中的推理参数提取到新字段。
3. **向量集合兼容性**：向量集合（Qdrant collection）已按 `tenant_{tenantId}_dataset_{datasetId}` 命名，改造后命名方式不变，向量数据无需迁移。
4. **EmbeddingModel 维度一致性**：如果历史数据使用的 embedding 维度与新配置不同（例如从 1024 改为 2048），需要重新向量化所有知识库内容，否则相似度搜索结果将不正确。建议保持维度配置不变。
5. **API 兼容性**：移除 `modelId` 参数后，前端聊天请求不再需要传入 `modelId`，需确认前端代码同步更新。
